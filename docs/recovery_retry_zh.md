# 转写失败自动重试（PR-56）

> 面向用户的简要说明。如需设计背景，参见
> `docs/superpowers/specs/2026-07-16-transcribe-resilience-retry.md`。

## 这一次改动对你意味着什么

之前，如果上一次崩溃后留下的 `audio_chunk_*.mp4` 不能被合并回 `audio.mp4`，
你只在控制台日志里能看到一行 `warn!`，应用本身没有任何提示，**你不知道恢复
失败了**。

PR-56 之后：

- 恢复过程在后台异步运行。你点了一次 [恢复] 后，可以正常用其他功能，**不会卡**
- 失败信息被持久化到 `<app_data>/recovery-state.json`。下次启动仍然能看到
- 顶部出现一条可展开的红色横幅：**"X 个会议恢复失败"**
- 展开后能看到每个失败的会议名 + 错误类型 + FFmpeg stderr 最后 500 字节
- 一键 [重试]（再次跑合并）或 [放弃]（从横幅移除）

## 重试策略

后台自动跑 **最多 3 次**，退避间隔 100ms / 500ms / 2s。3 次都失败后写入
持久化文件，由横幅兜底。

这样做是因为 FFmpeg 偶发失败（编码器临时不可用、文件被锁等）通常 1-2 次
重试就够了；永久失败（磁盘满、文件损坏）则由你介入。

## 横幅操作

横幅里的按钮：

- **重试**：立刻触发后台 3 次重试
- **放弃**：把这条记录标记为 discarded，下次启动不再提醒
- **查看日志**：展开 FFmpeg stderr 尾部

横幅在所有失败都被放弃后才隐藏。横幅存在期间不影响录音 / 转写 / 任何其他
功能——它只是 UI 一条提示条。

## 已知限制

1. `.checkpoints/` 里如果 chunk 物理损坏（中间帧缺数据），FFmpeg concat 会
   报错且**无法**自动修复——只能 [放弃] 并手动检查源音频是否可恢复
2. 持久化文件 IO 失败（磁盘满）仅记 warn log，下次启动重新扫描时会再次提示
3. stderr 只截 500 字节；如果你需要完整 FFmpeg 日志，请开一个 Issue 并附上
   会议名，我们再补扩展
4. 重试用的是 tokio Mutex 串行化（同一会议不并发），不同会议可并行

## 如果横幅"误报"怎么办

横幅里的 "重试" 按钮会重新发起恢复。如果初次失败只是临时 IO，重试多半成功。
点 [放弃] 仅是标记为"已知悉"——不会删除你的 `.checkpoints/` 或 `audio.mp4`
（这两者由 [清理] 按钮或重启管理）。
