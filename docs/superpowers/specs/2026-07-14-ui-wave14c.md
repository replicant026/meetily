## Background

PR-44a (shipped) added keyword highlighting. Now make the timestamp on
every transcript segment clickable so users can jump straight to the
audio playback position.

Meetily already has a `useAudioPlayer` hook with `seek(time)` exposed
but it has not been wired to the UI yet. PR-44c connects the dots.

## Goals

- Click a segment timestamp to seek audio playback to that position.
- TranscriptSegment accepts optional `onTimestampClick(sec: number)` prop.
- TranscriptPanel exposes `onTimestampClick` + optional `audioPath`.
- When `audioPath` is provided, the panel renders a compact audio
  player (play/pause + progress bar + time) using useAudioPlayer.
- 4 locale i18n for player controls.

## Non-Goals

- Audio file lookup logic (caller passes `audioPath`, PR-44d can wire
  meeting-folder auto-discovery)
- Waveform visualization
- Speaker-relative time (always uses absolute seconds in the recording)

## Design

```
TranscriptPanel.props.audioPath --> useAudioPlayer(audioPath)
                                  --> { isPlaying, currentTime, duration, seek, togglePlay }
                                                          |
TranscriptSegment.onTimestampClick (sec) -----------------+
                                                          v
                                                  audioPlayer.seek(sec)
```

If `audioPath` is null, no player UI is rendered; `onTimestampClick`
becomes a no-op (graceful).

## Risks

| Risk | Mitigation |
|---|---|
| audioPath is null in older meetings | Player UI hidden, click no-op |
| useAudioPlayer lazy-loads heavy data | Caller controls when to mount (audioPath gating) |
| Click vs scroll conflict | Button stops propagation |
