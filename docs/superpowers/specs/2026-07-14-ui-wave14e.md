# Wave 14: Parallel WAV Export (PR-44e)

> **Base branch:** feature/stability-wave8
> **Parent:** Wave 14 PR-44d (audio path discovery), shipped

## Background

PR-44d shipped the click-to-jump infrastructure plus a `get_meeting_audio_path`
Tauri command that probes for `audio.wav` in the meeting folder. PR-44d
intentionally returns `None` for mp4-only meetings, since the browser's Web
Audio API cannot decode AAC-in-MP4.

PR-44e closes that gap: whenever the recording pipeline writes the canonical
`audio.mp4`, it now also writes a parallel `audio.wav` next to it. The wav is
encoded as 16-bit PCM at the source sample rate, which `decodeAudioData` can
decode directly. From PR-44e onward, **every meeting auto-lights the player**.

## Goals

- On `finalize()`, after merging checkpoints into `audio.mp4`, also export a
  parallel `audio.wav` via ffmpeg transcode.
- The transcode is **non-fatal**: failure logs a warning but does not block
  finalize. `audio.mp4` remains the canonical recording.
- Disk cost ~10 MB/hour per recording (PCM 16-bit at 48 kHz stereo).

## Non-Goals

- Replacing `audio.mp4` with `audio.wav` (mp4 stays as the ASR input format).
- Real-time / live wav streaming during recording (only finalized output).
- Per-device tracks in the wav (mic / system / mix) — single mono/stereo mix only.
- Browser-side mp4 decoding.

## Design

```
incremental_saver::finalize()
  ├─ merge_checkpoints()      → audio.mp4      (existing, unchanged)
  ├─ transcode_mp4_to_wav()    → audio.wav      (new, non-fatal)
  └─ remove checkpoints dir
```

ffmpeg invocation:
```
ffmpeg -y -i audio.mp4 -c:a pcm_s16le -vn audio.wav
```

- `-c:a pcm_s16le`: 16-bit signed PCM little-endian (Web Audio API standard).
- `-vn`: defensive — drop any video track the mp4 container may carry.
- `-y`: overwrite (idempotent on re-finalize).

Sample rate and channel count follow the source (mp4 keeps its original
48 kHz stereo). `decodeAudioData` accepts any PCM sample rate/channel
combination the browser supports, so no resample is needed.

## Failure modes

| Scenario | Behavior |
|---|---|
| `ffmpeg` not installed | `transcode_mp4_to_wav` returns Err; finalize logs warn and proceeds with mp4-only |
| ffmpeg exits non-zero | Returns Err with stderr; finalize logs warn |
| Output file not created | Returns Err; finalize logs warn |
| mp4 missing (impossible after merge) | Returns Err; finalize logs warn |

In every failure case the canonical `audio.mp4` is still produced and
the recording is considered successful.

## Risks

| Risk | Mitigation |
|---|---|
| Disk usage doubles per meeting | Acceptable: ~10 MB/hour for wav, users opt in to recording anyway |
| Finalize takes longer (~few seconds for transcode) | Already happens at stop time, not in recording hot path |
| ffmpeg may not be bundled on all platforms | Existing `find_ffmpeg_path` auto-installs; same fallback applies |
| Old meetings without wav | `get_meeting_audio_path` (PR-44d) returns None; UI stays hidden |
