# Wave 14: Audio Path Discovery (PR-44d)

> **Base branch:** feature/stability-wave8
> **Parent:** Wave 14 PR-44c (clickable timestamp), shipped

## Background

PR-44c shipped the click-to-jump infrastructure (timestamp button,
`onTimestampClick` prop, compact audio player UI) but left
`audioPath` explicitly optional. Callers were expected to pass a
path; in practice nothing does. PR-44d wires the data layer so the
player activates automatically for any meeting that has a
browser-decodable audio file on disk.

## Format constraint (root cause)

Meetily records audio as AAC-in-MP4 (`audio/encode.rs`, `-c:a aac -f mp4`).
The browser's Web Audio API `decodeAudioData` does not support AAC
decoding. The `useAudioPlayer` hook therefore cannot decode the
primary recording even when the path is known.

**Workaround (this PR):** the backend lookup probes for `audio.wav`
in the meeting folder and returns the absolute path. Meetings with
only `audio.mp4` resolve to `null`, and the frontend stays silent.

**Long-term fix (future PR-44e):** record a parallel WAV stream
alongside MP4 so the browser can decode without re-encoding. Disk
cost ~10 MB/hour per recording.

## Goals

- Backend command `get_meeting_audio_path(meeting_id)` returns the
  absolute path to the meeting's WAV audio (when present) or `null`.
- Frontend hook `useMeetingAudioPath(meetingId)` exposes the resolved
  path along with `loading` / `error` state.
- `meeting-details/page-content.tsx` calls the hook and forwards
  the path to `TranscriptPanel` so click-to-jump just works.
- i18n parity across all 6 locales (en-US, en-GB, zh-CN, zh-TW,
  ja-JP, ko-KR).

## Non-Goals

- **WAV export at recording time** — PR-44e.
- **Browser-side MP4 decoding** — would require ffmpeg.wasm or a
  Tauri-side PCM conversion command.
- **Multi-track audio selection** (mic / system / mix) — always
  uses the single recording file.

## Design

```
meeting-details/page-content.tsx
  useMeetingAudioPath(meeting.id)
        |
        v
  invoke<Option<String>>("get_meeting_audio_path", { meetingId })
        |
        v
  Tauri -> src-tauri/src/database/commands.rs::get_meeting_audio_path
        |
        v
  AppState.db_manager.pool() -> MeetingsRepository::get_meeting_metadata(id)
        |
        v
  meeting.folder_path.join("audio.wav") -> exists? -> absolute path | null
```

If `audio.wav` does not exist, the command returns `Ok(None)`. The
frontend hook stores `null` and `TranscriptPanel` (with its existing
optional `audioPath` prop) hides the player UI.

## Risks

| Risk | Mitigation |
|---|---|
| mp4-only meetings return `null` forever | PR-44e ships WAV export |
| Backend DB not yet initialized when called | Returns `Ok(None)` cleanly (meeting not found) |
| Path canonicalization fails on Windows long paths | Falls back to un-canonicalized path |
| Locale parity broken for ja-JP / ko-KR | PR-44c missed them; this PR adds missing keys |
