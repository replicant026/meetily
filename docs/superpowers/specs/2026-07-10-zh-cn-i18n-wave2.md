## 1. Background

Wave 1 delivered the next-intl infrastructure, 12 locale JSON files, a `LocaleProvider`, and a `NextIntlClientProvider` wrapper in `layout.tsx`. The frontend hooks are wired, but `getStoredLocale()` and `setLocale()` currently swallow Tauri errors and fall back to `DEFAULT_LOCALE` because the Rust commands do not exist yet. Settings changes are lost on reload.

Wave 2 ships the persistence layer and a user-facing language picker so users can actually switch languages and have it survive an app restart.

## 2. Goals & Non-Goals

**Goals:**
- Persist the user's chosen UI language across app restarts via Tauri plugin-store.
- Expose `get_ui_language` and `set_ui_language` Tauri commands that mirror the onboarding module's pattern.
- Add a `LanguagePickerPopover` UI component and place it on the Settings > General page.
- Remove the try/catch silent fallbacks in `request.ts` and `useLocale.tsx` now that the commands exist.

**Non-Goals (deferred to later waves):**
- Migrating the remaining 200+ hardcoded English strings (Wave 3).
- Supporting `pt-BR`, `ja-JP`, etc. (Wave 4 / future).
- OS-locale auto-detect at first launch (Wave 4).
- Persisting the per-meeting summary language vs. the UI language (orthogonal; covered by `LANGUAGE_PREFERENCE` already in `lib.rs`).

## 3. Architecture

### 3.1 Persistence Layer (Rust)

Mirror the `onboarding.rs` pattern exactly:

- New file `src-tauri/src/i18n.rs`
- Store file: `ui-locale.json` (in app data dir, managed by `tauri-plugin-store`)
- Store key: `language` (string, e.g. `"en-US"`, `"zh-CN"`)
- No DB migration needed (plugin-store is file-based)

**Two Tauri commands:**

```rust
#[tauri::command]
pub async fn get_ui_language<R: Runtime>(app: AppHandle<R>) -> Result<Option<String>, String>;

#[tauri::command]
pub async fn set_ui_language<R: Runtime>(app: AppHandle<R>, language: String) -> Result<(), String>;

#[tauri::command]
pub async fn reset_ui_language<R: Runtime>(app: AppHandle<R>) -> Result<(), String>;
```

**Validation:** Reject unsupported locales with an error string. The frontend already ships the supported list in `frontend/src/i18n/config.ts`, but the Rust side is the source of truth at write time to defend against direct command invocations.

**Supported locale whitelist (Rust):** `["en-US", "zh-CN"]`. Kept in `src-tauri/src/i18n.rs` as a `pub const SUPPORTED_LOCALES: &[&str]`. A unit test asserts the list matches the frontend config (manual cross-check at PR review time).

### 3.2 Frontend Wiring

- Remove the `try { ... } catch { return DEFAULT_LOCALE }` swallows in `frontend/src/i18n/request.ts` `getStoredLocale()` and `frontend/src/hooks/useLocale.tsx` `setLocale()`. Errors should propagate to the caller (the layout for get; the UI for set).
- Add a single source-of-truth supported-locales list in `frontend/src/i18n/config.ts` that already exists; no new module needed.

### 3.3 `LanguagePickerPopover` Component

- New file `frontend/src/components/LanguagePickerPopover.tsx` ("use client").
- Uses existing `Popover` + `Command` (cmdk) UI primitives — no new design system work.
- Renders the current locale (e.g. "English (US)") as the trigger.
- Clicking a locale calls `useLocale().setLocale(next)` which now persists to the Rust store.
- Keyboard accessible: type-to-search the locale names.

### 3.4 Settings Integration

- Place the picker at the top of `PreferenceSettings.tsx` (the General tab content), as a new "Interface Language" card.
- Use `settings.preference.title` / `settings.preference.language` keys from `locales/<locale>/settings.json`.
- Localize the locale display names too (e.g. "简体中文", "English (US)"). Add 2 keys per locale: `settings.preference.locale_name_en_us` and `settings.preference.locale_name_zh_cn`.

## 4. Data Flow

```
On app launch:
  layout.tsx (RSC) -> getStoredLocale() -> invoke("get_ui_language")
    -> Tauri store file ui-locale.json -> "zh-CN"
    -> NextIntlClientProvider locale="zh-CN" messages=zh-CN
    -> <LocaleProvider initial="zh-CN">
    -> <html lang="zh-CN">

On user click in picker:
  LanguagePickerPopover -> useLocale().setLocale("en-US")
    -> invoke("set_ui_language", { language: "en-US" })
    -> Tauri store file ui-locale.json -> "en-US"
    -> setLocaleState("en-US") -> re-render with en-US messages
```

## 5. Failure Modes

| Failure | Behavior |
|---|---|
| Store file missing/corrupt | `get_ui_language` returns `Ok(None)`; layout falls through to next-intl's default-locale path (current Wave 1 behavior) |
| `set_ui_language` called with unsupported locale | Returns `Err("Unsupported locale: <x>")`; UI shows error toast; state does not change |
| Store write fails (disk full / permission) | Returns `Err`; UI shows error toast; previous in-memory locale preserved |
